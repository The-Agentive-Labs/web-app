Design Guide

Document design principles, patterns, and components to ensure consistency and scalability across your product.

Learn

What is a Design Guide?

A Design Guide (also called a Design System or Style Guide) is a comprehensive reference document that codifies your product's visual language, interaction patterns, and component library. It's the single source of truth that ensures designers, developers, and stakeholders create consistent, on-brand experiences. Think of it as your product's design DNA.

Why Create a Design Guide?

  • Consistency: Ensures UI looks and behaves the same across features, platforms, and teams.
  • Efficiency: Designers and developers reuse components instead of reinventing them.
  • Scalability: New team members onboard faster with clear documentation.
  • Quality: Pre-tested, accessible components reduce bugs and improve UX.
  • Brand Coherence: Visual identity remains cohesive as the product grows.

Core Components of a Design Guide

  • Design Principles: High-level values that guide decisions (e.g., "Simple over complex")
  • Color Palette: Primary, secondary, neutral, semantic colors (success, error, warning)
  • Typography: Font families, scales, weights, line heights
  • Spacing & Layout: Grid systems, spacing tokens, breakpoints
  • Components: Buttons, forms, cards, modals, navigation—with code examples
  • Patterns: Common UI patterns (empty states, error handling, loading states)
  • Accessibility: WCAG guidelines, contrast ratios, keyboard navigation
  • Voice & Tone: Writing guidelines for UI copy

Maturity Levels

  • Level 1: Static Style Guide: PDF or Figma file with colors, fonts, and basic components
  • Level 2: Living Documentation: Web-based guide with interactive examples and code snippets
  • Level 3: Component Library: Reusable, versioned components in code (React, Vue, etc.) synced with design tools
  • Level 4: Design System Team: Dedicated team maintaining, evolving, and supporting the system

Best Practices

  • Start Small: Begin with colors, typography, and 5-10 core components. Expand as needed.
  • Make It Accessible: Host it where everyone can access it (web, not just Figma or code repo).
  • Show, Don't Just Tell: Include visual examples and live code demos, not just descriptions.
  • Version Control: Track changes. Communicate updates to teams when components change.
  • Include Do's and Don'ts: Show correct and incorrect usage with examples.
  • Make It Searchable: Teams should find what they need in under 30 seconds.
  • Keep It Alive: Design guides aren't one-and-done. Update regularly as the product evolves.

Build

Template: Design Guide Structure

1. Introduction
- Purpose of this guide
- Design principles (3-5 core values)

2. Foundation
- Color palette (primary, secondary, neutral, semantic)
- Typography (font families, scales, weights)
- Spacing system (tokens: 4px, 8px, 16px, etc.)
- Grid & layout (columns, breakpoints)

3. Components
For each component:
- Visual example
- Variants (default, hover, disabled, etc.)
- Usage guidelines (when to use, when not to use)
- Code snippet (HTML/CSS/React/etc.)
- Accessibility notes

4. Patterns
- Form validation
- Empty states
- Loading states
- Error handling

5. Voice & Tone
- Writing guidelines
- Example microcopy

Step-by-Step Instructions

  1. Audit Existing Design: Take inventory of all UI elements currently in your product. What's consistent? What's inconsistent?
  2. Define Design Principles: Write 3-5 core values (e.g., "Clarity over cleverness," "Fast and lightweight"). These guide all decisions.
  3. Document Foundation: Extract color palette, typography, and spacing from existing designs. Create semantic tokens (e.g., `color-primary`, `spacing-md`).
  4. Catalog Components: List all UI components (buttons, inputs, cards, etc.). For each, capture variants, states, and usage rules.
  5. Build Examples: Create visual examples and code snippets. Show components in context (e.g., button in a form).
  6. Add Accessibility: Document contrast ratios, keyboard navigation, ARIA labels. Test with screen readers.
  7. Publish & Share: Host on a web platform (Storybook, Zeroheight, custom site). Share link with all teams.
  8. Iterate: Gather feedback. Update as you design new features. Version major changes.

Example: Freelance Invoice App Design Guide

Design Principles:

  • 1. Simple First: Prioritize clarity over features. If it requires explanation, simplify.
  • 2. Fast & Light: Interactions should feel instant. No unnecessary animations or bloat.
  • 3. Human, Not Corporate: Friendly, conversational tone. Reduce anxiety, not increase it.

Color Palette:

  • Primary: Purple #963BF7 (CTAs, links, highlights)
  • Success: Green #10B981 (paid invoices, success states)
  • Warning: Orange #F59E0B (overdue invoices)
  • Error: Red #EF4444 (errors, destructive actions)
  • Neutral: Gray scale from #050506 (text) to #FAFAFA (backgrounds)

Typography:

  • Headings: Geist, 700 weight
  • Body: Google Sans, 400 weight
  • Code/Numbers: JetBrains Mono, 400 weight
  • Scale: 12px (small), 14px (body), 16px (default), 20px (subheading), 24px+ (headings)

Component Example: Primary Button

  • Default: Purple background (#963BF7), white text, 8px vertical padding, 16px horizontal padding, 8px border radius
  • Hover: Darken to #7B2FD8
  • Disabled: Gray background (#DDD8DD), gray text (#C9C4CC)
  • Usage: Use for primary actions (Send Invoice, Save, Create). Limit to 1 per screen.
  • Don't: Use for destructive actions (Delete). Use error-styled button instead.

Pattern Example: Invoice Status

  • Pending: Orange dot + "Pending" text
  • Paid: Green checkmark + "Paid" text + celebration animation
  • Overdue: Red warning icon + "Overdue by X days" text

AI + Human Collaboration

Human Value

  • Strategic vision to define design principles that align with brand
  • Design judgment to create cohesive visual systems
  • Contextual understanding of when to break the rules
  • Accessibility expertise to ensure inclusive design

AI Value

  • Automated extraction of colors and typography from existing designs
  • Generation of component variants and states
  • Documentation drafting with code examples
  • Contrast ratio calculations and accessibility checks

Key Considerations

  • AI for Extraction: Use AI to scan existing designs and extract color palettes, spacing patterns, and typography scales. Humans curate and refine into semantic tokens.
  • Component Variants: AI can generate common component states (hover, disabled, focus). Humans ensure they match brand and accessibility standards.
  • Documentation Drafting: AI can draft component documentation and code snippets. Humans add usage guidelines, do's/don'ts, and context.

"AI in the Loop" Model

  1. Audit: AI scans Figma files or codebase to extract design tokens (colors, fonts, spacing). Humans review and organize into a semantic system.
  2. Component Documentation: AI generates initial component specs (props, variants, code examples). Humans add usage guidelines and accessibility notes.
  3. Accessibility Checks: AI calculates contrast ratios and flags WCAG violations. Humans adjust colors and document keyboard navigation patterns.

Watch-Outs

  • Don't let AI define your design principles. They require strategic brand alignment and human judgment.
  • Avoid AI-generated component names. They may be generic ("Button1", "Card2"). Use semantic names ("PrimaryButton", "InvoiceCard").
  • Be cautious of AI accessibility checks. They catch contrast and structure issues but miss semantic HTML and screen reader UX.
  • Don't auto-generate the entire design guide. It needs human curation, examples, and storytelling to be useful.
